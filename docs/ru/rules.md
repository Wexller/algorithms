# Правила работы

## Общие правила

- Вся основная работа делается локально.
- Нельзя помечать задачу как `solved`, если тесты пропущены или заметки не заполнены.
- Нельзя переходить к следующему обязательному модулю, если текущий не закрыт по required-задачам.
- Нельзя коммитить заведомо сломанное решение.

## Правила по решению

- Сначала формулируй идею на естественном языке.
- Только потом пиши код.
- После кода обязательно фиксируй сложность по времени и памяти.
- Если первая идея не сработала, это должно появиться в `attempts.md`.
- Если решение опирается на паттерн, назови его явно.

## Правила по проверке

- Каждый solved-проблем должен иметь хотя бы один основной тест и набор крайних случаев.
- Для структур данных и графов должны быть тесты на пустой или минимальный вход.
- Для массивов и строк должны быть тесты на повторы и границы.
- Если задача допускает брутфорс на маленьком входе, желательно добавить oracle-проверку.

## Правила статусов

- `pending` — задача не доведена до рабочего стандарта.
- `blocked` — ты понимаешь формулировку, но застрял и можешь описать, где именно.
- `solved` — решение, тесты и заметки готовы, `npm run check` проходит.

## Правила коммитов

- Один коммит должен отражать один логический шаг.
- Если меняешь решение, меняй и тесты при необходимости.
- Коммиты `study:` используются для новых решенных задач.
- Коммиты `fix:` используются для исправления логики, тестов и валидаторов.
